# project-catchup

途中参加したプロジェクトの全体像を、**構造化されたテーブル**で短時間に把握するためのskill。

## 使用タイミング

- 新しく配属された/引き継いだプロジェクトの把握
- 既存プロジェクトの監査・レビュー着手時
- オンボーディング資料の下書き作成

## 調査手順

以下の順で実施する。調査対象が見つからなかった項目は「不明（要ヒアリング）」と明記し、推測で埋めない。

1. **ルート直下の確認** — `README.md`, `CLAUDE.md`, `docs/`, `ARCHITECTURE.md`, `CONTRIBUTING.md`
2. **依存関係とランタイム** — `package.json`, `pyproject.toml`/`requirements.txt`, `Cargo.toml`, `go.mod`, `Gemfile`, `composer.json`, `Dockerfile`, `docker-compose.yml`
3. **インフラ・デプロイ** — `.github/workflows/`, `terraform/`, `k8s/`, `serverless.yml`, `vercel.json`, `netlify.toml`, `fly.toml`, `railway.toml`
4. **ディレクトリ構造の把握** — 主要ディレクトリを列挙（深さ2〜3まで）
5. **git履歴** — `git log --since="3 months ago" --pretty=format:"%h %an %s" | head -50`、`git shortlog -sn --since="6 months ago"`（アクティブな貢献者）
6. **Issue/PR** — `gh issue list --state open --limit 20`、`gh pr list --state open`（gh利用可能時）
7. **設定ファイル** — `.env.example`, `config/`（外部サービス連携の痕跡）
8. **不足情報のヒアリング** — 下記表の「不明」項目をまとめてユーザーに提示

## 出力時の注意

- **推測と事実を区別する** — 推測には「（推測）」を付ける。
- **空欄を残さない** — 情報が取れなかったら「不明（要ヒアリング）」か「該当なし」と書く。
- **長文にしない** — 各セルは原則1〜2行。詳細が必要なら別途箇条書きで補足。
- **最後にヒアリング項目リストを提示** — 不明欄がある場合、ユーザーに質問すべき項目を箇条書きでまとめる。
